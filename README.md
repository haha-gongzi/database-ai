# DatabaseAI Project

## System Overview

This project implements a simplified AI-assisted database system using Python and SQLite.

The system supports:

* Loading CSV files into a database
* Automatic schema inference
* SQL query validation (security layer)
* Controlled query execution
* Command-line interface (CLI)

### Architecture

CLI → QueryService → SQLValidator → Database
↑
SchemaManager
↑
CSVLoader

## Features

### CSV Loader

* Uses pandas to read CSV files
* Automatically creates database tables
* Infers schema from data types

### Schema Manager

* Normalizes column names
* Tracks table schemas
* Ensures consistency across queries

### SQL Validator (Security Layer)

* Only allows SELECT queries
* Blocks dangerous operations (DELETE, DROP, UPDATE, etc.)
* Prevents SQL injection
* Validates table existence

### Query Service

* Central query execution layer
* Ensures validation before execution
* Returns structured results

### CLI Interface

Available commands:

```text
load <csv_path> <table_name>
tables
schema <table_name>
query <SQL>
exit
```

## Notes

* SQLite is used as the database backend
* pandas is used only for CSV ingestion
* All SQL queries are validated before execution for security

## Design Decisions

The system follows a modular architecture. The CLI does not access the database directly. Instead, it sends requests to the Query Service. The Query Service validates SQL using the SQL Validator before executing anything on the database.

This design separates user interaction, validation, database access, and LLM-based SQL generation. It also makes each component easier to test independently.

## LLM Usage

The LLM Adapter is used only to convert natural language questions into SQL queries. The LLM does not execute SQL directly. All SQL generated by the LLM is treated as untrusted input and must pass through the SQL Validator before execution.

## SQL Validation

The SQL Validator protects the database by enforcing three rules:

1. Only SELECT queries are allowed.
2. Queries referencing unknown tables are rejected.
3. Queries referencing unknown columns are rejected.

## Testing Strategy

Unit tests were written for the SQL Validator, Query Service, CSV Loader, and Schema Manager. The tests check valid queries, invalid queries, CSV ingestion, schema creation, and safe rejection of dangerous SQL.

One important test checks that an invalid SQL query referencing an unknown column is rejected. This represents a case where an LLM may generate incorrect SQL, and the validator catches the mistake before execution.

## Limitation

The current SQL Validator supports basic SELECT queries, but it does not fully parse complex SQL such as joins, aliases, nested queries, or aggregate expressions. A future improvement would be to use a more complete SQL parsing library while still keeping strict validation rules.
