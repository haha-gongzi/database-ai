# DatabaseAI Project

## System Overview

This project implements a simplified database query system with an LLM-inspired SQL generation layer using Python and SQLite.

The system supports:

* Loading CSV files into a database
* Automatic schema inference
* SQL query validation
* Controlled query execution
* Command-line interface (CLI)

---

## Architecture

```text
CLI
 ↓
QueryService
 ↓
SQLValidator
 ↓
Database

CSVLoader
 ↓
SchemaManager
```

The system follows a modular architecture where each component has a separate responsibility.

The CLI never directly accesses the database. All database operations go through the Query Service and SQL Validator layers.

## Features

### CSV Loader

* Uses pandas to read CSV files
* Automatically creates database tables
* Infers schema from CSV data types

### Schema Manager

* Normalizes column names
* Tracks database schemas
* Ensures schema consistency

### SQL Validator

* Only allows SELECT queries
* Rejects dangerous operations such as DELETE, DROP, and UPDATE
* Rejects queries referencing unknown tables
* Rejects queries referencing unknown columns

### Query Service

* Central query execution layer
* Ensures validation before execution
* Returns structured query results

### LLM Adapter

* Converts natural language requests into SQL queries
* Treats generated SQL as untrusted input
* Sends generated SQL to the validator before execution

### CLI Interface

Available commands:

```text
load <csv_path> <table_name>
tables
schema <table_name>
query <SQL>
exit
```

### Continuous Integration

* GitHub Actions automatically runs tests on every push
* Helps ensure repository stability
* Detects failures early during development

## Design Decisions

The project follows a modular architecture based on separation of concerns.

The CLI handles user interaction only.

The Query Service is responsible for query execution.

The SQL Validator validates SQL queries before execution.

The Database layer handles SQLite operations.

The LLM Adapter only generates SQL and never directly executes queries.

This layered design improves maintainability, security, and testability.

---

## LLM Usage

The LLM Adapter is used only to convert natural language questions into SQL queries.

The generated SQL is always treated as untrusted input.

All generated SQL must pass through the SQL Validator before execution.

The LLM never directly accesses the database.

---

## SQL Validation

The SQL Validator protects the system by enforcing the following rules:

1. Only SELECT queries are allowed.
2. Queries referencing unknown tables are rejected.
3. Queries referencing unknown columns are rejected.

This helps prevent unsafe or invalid database operations.

---

## Testing Strategy

Unit tests were implemented for:

* SQL Validator
* Query Service
* CSV Loader
* Schema Manager

The tests verify both valid and invalid behaviors.

Examples include:

* Valid SELECT query execution
* Rejecting DELETE queries
* Rejecting unknown tables
* Rejecting unknown columns
* CSV ingestion and schema creation

One important test simulates incorrect SQL generated by an LLM. The validator catches the issue before execution.

---

## Limitation

The current SQL Validator supports basic SELECT queries only.

It does not fully support advanced SQL features such as:

* JOIN queries
* Aliases
* Nested queries
* Aggregate expressions

Future improvements could include using a more advanced SQL parsing library while still maintaining strict validation rules.

---

## Notes

* SQLite is used as the database backend
* pandas is used only for CSV ingestion
* All SQL queries are validated before execution
